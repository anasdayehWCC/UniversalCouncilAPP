# Session: Development Automation Overhaul — 2026-04-18

## What Happened

Cross-project audit comparing grants-platform `.claude/` with UniversalCouncilAPP `.claude/`, followed by targeted improvements. Started by porting 3 missing capabilities (brainstorming, writing-plans, orchestrator-claims), then performed a deep analysis of systemic problems in how the automation system operates.

## Key Findings (from deep analysis)

### Problem 1: Busywork over impact
- 32% of production-backlog items were minor a11y (aria-labels, focus indicators)
- 3 CRITICAL items (admin persistence, audit trail, tenant onboarding) sat untouched across multiple orchestration runs
- Orchestrator gravitates toward bounded lint fixes because they're easy to close

### Problem 2: One agent per tiny task
- 3 separate ui-engineers dispatched for work one agent could handle sequentially
- Grouping was by file directory independence, not domain relatedness
- Massive context-loading overhead per agent

### Problem 3: No visual assessment
- All quality gates were text-based linters (grep, eslint, audit scripts)
- No evidence of screenshots, visual diffs, or "does it look right?" in any session
- System would ship broken layouts with perfect lint scores

### Problem 4: No product vision
- 41-phase roadmap but zero words on "why" or "for whom" or "what premium means"
- Agents couldn't make trade-off decisions without knowing the product goal
- Prioritisation defaulted to "unblock the build" not "deliver value"

### Problem 5: Context-starved sub-agents
- Briefs contained file paths + technical rules but NOT product context, design intent, data contracts, or success criteria beyond lint
- Agents built technically correct but potentially wrong-from-product-perspective code

### Problem 6: Zero housekeeping consciousness
- No refactoring passes, no component extraction, no stale code cleanup
- Discovered issues handled as one-offs, never systematically

## Learnings

### Trigger: System fixes minor issues while critical features remain mock-only
Rule: Enforce the prioritisation matrix from product-vision.md. At least one work package per orchestrator run MUST address priority categories 1-3 (ship-blockers, core flows, integration) if any exist. Never dispatch all agents on category 5-6 (polish/compliance) work.
Verify: Check production-backlog.md for CRITICAL items before dispatching; session notes record which priority categories were addressed.

### Trigger: Multiple agents dispatched for isolated single-task work
Rule: Batch by domain area (recording, review, admin, notes, platform), not by file independence. One agent with 3-5 related tasks builds more coherent code than 5 agents with 1 task each. Max 3 work packages per run.
Verify: Each dispatched agent receives a multi-task work package brief, not a single ticket.

### Trigger: UI work merged without visual verification
Rule: Visual checkpoint is mandatory before merge. Start dev server, open affected routes at desktop and mobile viewports, check against design principles. Lint passing ≠ visual quality.
Verify: Phase 5 of orchestrator includes visual checkpoint section; agent reports include "Visual verification" description.
